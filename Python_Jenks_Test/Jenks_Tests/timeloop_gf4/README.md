# GF4-Engine — Timeloop + Accelergy model

Component-based energy **and area** model for the imaginary GF4-Engine
accelerator (the hardware that runs our Gaussian-codebook GF4 W4A4 natively).
Upgrades the analytical `../hw_sim_gf4.py` roofline with a real Accelergy
component model and adds the **area** axis + the **LUT overhead** vs NVFP4.

## What it models
- **Architecture** (`arch/`): 16×16 = 256-PE weight-stationary array, DRAM →
  global buffer → per-PE regfiles → MAC.
  - `gf4_engine.yaml`  — PE = `lut_fp4_mac` (16-entry programmable codebook + MAC).
  - `nvfp4_engine.yaml` — PE = plain `intmac` (hardwired E2M1). Same everything else.
  - Running both isolates the **LUT energy + area cost** (our central HW claim:
    the LUT is nearly free).
- **Component** (`components/lut_fp4_mac.yaml`): the LUT-FP4 MAC compound — on a
  `mac_random`, 2× LUT decode (weight + activation code) + 1 int MAC.
- **Problems** (`prob/`): each unique OPT linear shape as a 1×1 conv
  (C=in, M=out, N=tokens=2048), generated by `gen_problems.py` with per-shape
  multiplicities in `manifest.yaml`.
- **Not in Timeloop** (added analytically in `postprocess.py`, ~few % each, per
  `hw_sim_gf4.py`): the Hadamard FWHT pre-pass, the per-16 microscale dequant,
  and the residual 2× factor — Timeloop schedules the GEMM, not pre-transforms.

## Larger networks: Colab → here round-trip
Run the iso-energy cell (cell 5) in `../Colab/FP_Quant.ipynb` on a big model
(set `MODEL`).  It writes `<model>_hwsim.json` (auto-downloads in Colab) holding
the ACTUAL per-layer (K,N) shapes + the measured A16/NVFP4/GF4/residual PPLs —
architecture-agnostic (Llama GQA, gated MLP, OPT all handled).  Then locally:
```bash
cp ~/Downloads/<model>_hwsim.json timeloop_gf4/models/
python3 gen_problems.py     # ingests models/*.json -> prob/ + manifest
./run.sh && python3 postprocess.py        # energy + area + LUT overhead
python3 ../hw_sim_gf4.py     # also auto-loads models/*.json -> accuracy/energy Pareto
```
`gen_problems.py` and `hw_sim_gf4.py` both auto-discover `models/*.json`, so the
new model appears in the Timeloop runs and the Pareto with no code edits.

## Run it
Docker is required. Your account is **not** in the `docker` group yet, so either:

```bash
# Option A (persistent — recommended): add yourself to the docker group, re-login
sudo usermod -aG docker $USER
newgrp docker            # or log out/in

# Option B (one-off): run everything through sudo
export DOCKER='sudo docker'
```

Then:
```bash
docker pull timeloopaccelergy/accelergy-timeloop-infrastructure:latest   # once
python3 gen_problems.py        # already done — writes prob/*.yaml + manifest.yaml
./run.sh                       # runs both engines over all shapes -> out/
python3 postprocess.py         # weighted totals: energy/token, area, LUT overhead
```

> In this session run the pull yourself with the `!` prefix, e.g.
> `! sudo docker pull timeloopaccelergy/accelergy-timeloop-infrastructure:latest`

## Output
`postprocess.py` reports, per model: GF4 vs NVFP4 GEMM energy/token, array
**area (mm²)**, the **LUT energy & area overhead %**, and the GF4 1-term /
residual energy with the FWHT+microscale add-ons.

## Notes / tunables
- Tech node is 45 nm (`technology:` in the arch files) to match the
  Horowitz-grounded `hw_sim_gf4.py`; switch to a predictive PDK later if porting
  the RTL path (Gemmini → OpenROAD/ASAP7).
- Array size (256 PE), buffer depth, LUT width are first-pass; sweep in the arch
  YAMLs.
- The schema here is Timeloop v0.3 (`subtree`/`local`), supported by the
  official infrastructure image. If the image ships only the v0.4 `!Container`
  loader, the arch files convert mechanically (same hierarchy).
